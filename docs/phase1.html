第一梯次：內部 MVP

目標期限：6月底前 | 重點：高 CP 值 Azure 基礎設施與內部營運

階段任務 核心目標與範圍

專注於內部核心功能，讓團隊能提早熟悉新系統，並確立最低成本的 Azure 雲端環境標準。

  • 基礎建設：配置 MongoDB Atlas 與 Azure Container Apps (ACA)，捨棄昂貴的 K8s 叢集。
  • 前端部署：全面採用 Azure Static Web Apps 免費層級。
  • 身分驗證 (Auth)：實作內部員工 JWT 登入模組。
  • 基礎營運：產品型錄、庫存盤點管理、揀貨單自動化與 POS 基礎設定。

潛在風險

  • 系統初期可能不穩定，導致內部測試受阻。
  • MongoDB Atlas 設定錯誤可能導致效能瓶頸或費用暴增。
  • 缺乏龐大的 API Gateway，初期需倚賴 ACA 內建 Ingress，管理較為陽春。

修復與改進

  • 第一版先開通 MongoDB Atlas 無伺服器模式 (Serverless) 或共享叢集以控制成本。
  • 利用 Azure Container Apps 的 Scale to Zero 特性，降低非營業時間的運算費用。
  • 嚴格落實 Azure DevOps CI/CD 自動化發布流水線，減少人工作業。

後端工程 Backend Backlog Tickets

[BE-101] 配置 MongoDB Atlas 與基礎 Collections
[基礎設施: MongoDB / Go]
設計並建置 MongoDB Atlas 的 Management 與 Operations 集合，包含員工 (Users, Roles) 與庫存 (SKUs, Inventories)。
驗收標準 (Acceptance Criteria):
  • 完成 MongoDB Migration 腳本或建立必要的 Index (如 Unique Index)，無報錯執行。
  • 完成關聯設計 (利用 ObjectId 參照如 `inventory_id` 對應 `sku_id`)。
[BE-102] 實作 Backend-Shared-Contract
[程式庫: Backend-Shared-Contract (Go)]
在獨立 Repository 定義全域使用的 Enums (如訂單狀態碼、角色列舉)、DTOs (Data Transfer Objects)。
驗收標準 (Acceptance Criteria):
  • 完成 Go 語言的 Struct 宣告，並加上適當的 JSON/BSON Tag。
  • 發布至私有 Git 庫或 Azure Artifacts 供其他微服務 Import。
[BE-103] 開發 JWT 身分驗證模組與 RBAC API
[微服務: Mgmt & Auth API (Go)]
實作員工登入端點 (`/api/v1/auth/login`)，發放 JWT Access Token。實作 RBAC 權限檢查 Middleware。
驗收標準 (Acceptance Criteria):
  • 密碼必須使用 bcrypt 進行雜湊加密儲存。
  • Middleware 能夠正確解析 Token 並拒絕無權限的存取 (回傳 401/403)。
[BE-104] 部署服務至 Azure Container Apps (ACA)
[基礎設施: ACA / DevOps]
捨棄昂貴的 AKS，改為撰寫 ACA Bicep 腳本或透過 Azure CLI，並在 Azure DevOps 建立 CI/CD 流水線。設定 Scale to Zero 以極小化成本。
驗收標準 (Acceptance Criteria):
  • Git Commit 推送至 main 分支後，能自動編譯 Docker Image 推送至 ACR，並成功更新 ACA 容器。
[BE-105] 開發揀貨單自動化產出與列印 API
[微服務: Operations API (Go)]
針對成立的訂單自動生成 PDF 或 HTML 格式的揀貨單 (Picking List)，支援批次匯出功能。
驗收標準 (Acceptance Criteria):
  • 能根據倉庫儲位排序揀貨清單，提升店員揀貨效率。
  • API 回應時間小於 500ms，產生的 PDF 需支援條碼/QR Code 掃描。

前端工程 Frontend Backlog Tickets

[FE-101] 搭建 potato-control 網頁後台框架
[前端專案: Potato-Control (Web)]
使用 Vite + React + TypeScript 初始化專案。配置 Tailwind CSS，設定 React Router 與全域狀態 (Tanstack Query)。
驗收標準 (Acceptance Criteria):
  • 成功執行 `npm run dev` 且無 TypeScript 錯誤。
  • 完成基礎的左側導覽列 (Sidebar) 與頂部列 (Header) Layout 元件。
[FE-102] 實作登入頁面與 Token 攔截器 (Interceptor)
[前端專案: Potato-Control (Web)]
實作登入表單 UI。設定 Axios/Fetch Interceptor，自動在後續 API Request 的 Header 加上 `Authorization: Bearer `。
驗收標準 (Acceptance Criteria):
  • 輸入錯誤帳密能正確顯示來自 API 的錯誤訊息。
  • 登入成功後能正確將 Token 存入 localStorage / Cookie，並跳轉至首頁。
[FE-103] 開發「庫存盤點」React 元件與表單
[前端專案: Potato-Control (Web)]
提供快速檢索 SKU 的 DataGrid (可搭配 Tanstack Table)，並實作單筆或批次修改庫存數量的對話框 (Dialog)。
驗收標準 (Acceptance Criteria):
  • 能夠透過名稱或 SKU 條碼即時搜尋商品。
  • 修改數量送出後，需有防呆確認視窗，並能顯示 API 成功或失敗的 Toast 提示。
[FE-104] 前端資源免費部署至 Azure Static Web Apps
[基礎設施: DevOps / Frontend]
設定 Azure Pipelines 將編譯後的 Vite 打包產物部署至 Azure Static Web Apps 的 Free 層級，享有零成本託管與全域快取。
驗收標準 (Acceptance Criteria):
  • 開發團隊 Push 至 Git 後能自動發布到 dev 環境網址，且具備 SSL 憑證。
[FE-105] 實作 POS 機基本設定與收銀綁定 UI
[前端專案: Potato-Control (Web & APP)]
在後台與 POS APP 實作收銀機綁定設定，包含硬體周邊 (掃碼槍、出單機) 的連線設定與基礎 UI。
驗收標準 (Acceptance Criteria):
  • 可透過藍牙或網域 IP 成功綁定熱感應印表機。
  • 具備「測試列印」功能按鈕。
