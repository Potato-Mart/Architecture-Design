第三梯次：雙平台網頁與手機版上線

目標期限：9月底前 | 重點：B2C 與 B2B 客戶端、原生行動 APP 雙平台上架

階段任務 核心目標與範圍

正式對一般消費者 (B2C) 與企業客戶 (B2B) 發布產品，涵蓋網頁版與原生行動應用程式 (iOS/Android)。

  • B2B 專屬平台：發布 Potato-Partner (Web & APP)，提供企業批量採購與階梯式定價。
  • 行動端發布：利用 Kotlin (Android) 與 Swift (iOS) 將 Potato-Store APP 提交至 App Store / Google Play 雙平台商店。
  • 推播系統：實作 Azure Notification Hubs 推送促銷通知與訂單狀態至用戶手機。

潛在風險

  • Apple App Store 審查可能因為隱私權或金流合規問題而遭退件，導致延遲上線。
  • Azure 輕量級 Ingress 面對大量 B2C 惡意請求或爬蟲時，缺乏進階 WAF 的保護。
  • 雙平台 (B2C 與 B2B) 會員資料打架或越權存取。

修復與改進

  • 提早至少兩週將 iOS 版本送審 TestFlight 進行合規測試。
  • 在此階段可考慮導入平價版的 Azure Front Door (Standard) 作為邊緣防護與 WAF，擋掉惡意流量。
  • 在 Auth API 嚴格區分 `Role` (消費者 vs 企業)，並在 ACA 的 API 路由層嚴格校驗 JWT。

後端工程 Backend Backlog Tickets

[BE-301] 開發 B2B 階梯式定價與審批 API
[微服務: Commerce API (Go)]
實作企業帳號專屬的定價邏輯，支援依採購數量動態計算折扣 (Tiered Pricing)，並實作企業內部子帳號的採購額度審批 (Approval Workflow)。
驗收標準 (Acceptance Criteria):
  • 企業購物車計算出的單價必須正確反映折扣階層。
  • 超過採購額度的訂單狀態必須進入 `PENDING_APPROVAL`。
[BE-302] 串接 Azure Notification Hubs 實作推播系統
[微服務: Ops / Auth API (Go)]
註冊行動端設備的 FCM (Android) 與 APNs (iOS) Token 至資料庫，並提供觸發推播通知的 API (例如：出貨通知)。
驗收標準 (Acceptance Criteria):
  • 能根據 UserID 精準推播至特定使用者的所有設備。
[BE-303] 強化 Auth API 區分 B2B 與 B2C 登入邏輯
[微服務: Auth API (Go)]
更新登入端點，支援社群登入 (Google/Apple) 給一般消費者，並為企業用戶加入嚴格的信箱網域驗證。
驗收標準 (Acceptance Criteria):
  • 成功核發夾帶 `role=B2B` 或 `role=B2C` 的 JWT。
[BE-304] Azure Front Door WAF 防護與快取設定
[基礎設施: Azure / DevOps]
在架構最外層部署 Azure Front Door (Standard)，開啟 Web Application Firewall (WAF) 阻擋 SQLi 與 XSS，並設定 API 回應快取。
驗收標準 (Acceptance Criteria):
  • 模擬 SQL Injection 攻擊測試需收到 403 Forbidden 回應。

前端工程 Frontend Backlog Tickets

[FE-301] 開發 Potato-Partner (Web) 企業版前台
[前端專案: Potato-Partner (Web)]
使用 Vite + React，套用沉穩的 B2B 專屬 Tailwind 企業色系。實作大宗採購購物車與審批進度條元件。
驗收標準 (Acceptance Criteria):
  • 提供一鍵帶入上次採購清單的功能。
  • 部署至 Azure Static Web Apps 免費層，並掛載 B2B 專屬網域。
[FE-302] 開發 Potato-Store 原生 APP 雙平台 (iOS/Android)
[行動端: Potato-Store (APP)]
使用 Kotlin (Android) 與 Swift (iOS) 打造流暢的原生購物體驗，整合本機推播接收。
驗收標準 (Acceptance Criteria):
  • 畫面切換需達 60fps 原生流暢度，無白畫面卡頓。
[FE-303] 實作 APP 內建 Apple Pay 與 Google Pay
[行動端: Potato-Store (APP)]
透過 Adyen Native SDK，於結帳頁面喚起原生的 Apple Pay (iOS) 與 Google Pay (Android) 快速結帳。
驗收標準 (Acceptance Criteria):
  • 點擊 Apple Pay 按鈕需正確跳出指紋/Face ID 驗證，並將 Token 回傳給後端 Commerce API。
[FE-304] 行動端 Fastlane CI/CD 自動發布腳本
[基礎設施: Mobile / DevOps]
撰寫 Fastlane 腳本 (Ruby)，結合 Azure DevOps Pipelines，實現自動編譯 APP 並推播至 TestFlight 與 Google Play Console 內部測試軌道。
驗收標準 (Acceptance Criteria):
  • 自動遞增 Build Number，開發人員不需手動上傳 IPA 或 AAB 檔案。
