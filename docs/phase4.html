第四梯次：效能強化與直播導入

目標期限：11月底前 | 重點：迎接年底旺季、直播帶貨 (方案A)、效能除錯

階段任務 核心目標與範圍

在年底的電商購物旺季 (如雙11、黑五) 前完成系統壓測，並上線「電商直播」功能以刺激銷量。

  • 全面檢視 Phase 1~3 的所有功能，進行深度除錯 (Bug fixing) 與效能優化。
  • 在網頁與 APP 中整合「電商直播 (Live Streaming)」功能，實現邊看邊買。
  • 利用 Azure Cache for Redis 快取與 MongoDB Atlas 索引優化，提升回應速度。

潛在風險

  • 高併發下單 (搶購) 可能導致超賣問題 (Overselling)。
  • 影片直播串流極耗流量，若自己架設流媒體伺服器，可能導致頻寬塞爆且成本難以控制。
  • 直播期間的大量即時聊天訊息，可能拖垮後端微服務。

修復與改進

  • 透過 Redis 分散式鎖 (Distributed Lock) 防止庫存超賣。
  • **採用「方案 A」**：直播串流與轉碼交由專業的外部 SaaS (如 Mux API 或是 AWS IVS) 處理，減輕主伺服器負擔。
  • 將聊天室微服務抽離成獨立的 Socket.io/Websocket Container App 叢集。

後端工程 Backend Backlog Tickets

[BE-401] 開發 Live Stream API (方案 A：串接 Mux / IVS)
[微服務: Live Stream API (Node.js)]
建立管理直播間的 API，向 Mux API 發出請求以取得 RTMP 推流網址與 HLS 播放網址。
驗收標準 (Acceptance Criteria):
  • 管理員建立直播間後，能正確回傳包含 Stream Key 的 RTMP 網址供 OBS 軟體使用。
  • 使用者請求時回傳有效的 HLS (.m3u8) 播放連結。
[BE-402] 實作即時 WebSocket 聊天室與購物車廣播
[微服務: Live Stream API (Node.js)]
使用 Socket.io 建立即時聊天伺服器。實作聊天室訊息的限流 (Rate Limiting) 以及「店長推播商品」的廣播事件。
驗收標準 (Acceptance Criteria):
  • 需在 Azure Container Apps 設定 Sticky Sessions 以支援 WebSocket 負載平衡。
  • 單一帳號發言頻率限制每秒不可超過 2 次，違規者丟棄訊息。
[BE-403] 實作高併發搶購之 Redis 分散式鎖
[微服務: Commerce API (Go)]
修改現有扣庫存邏輯，導入 Redis SETNX 鎖機制 (或 Redlock 演算法)，確保限量商品在雙十一期間不發生超賣。
驗收標準 (Acceptance Criteria):
  • 使用 JMeter 發動 1,000 併發購買限量 50 件商品，最終資料庫庫存量必須精準為 0，且只有 50 筆成功訂單。
[BE-404] MongoDB Atlas 慢查詢優化與索引調整
[基礎設施: MongoDB Atlas / Go]
檢閱過去一季的慢查詢紀錄 (Slow Queries)，補齊缺失的複合索引 (Compound Index)。
驗收標準 (Acceptance Criteria):
  • 所有 API 的 95th Percentile (P95) 回應時間必須小於 200ms。

前端工程 Frontend Backlog Tickets

[FE-401] 原生 APP 直播播放器 (HLS) 實作
[行動端: Potato-Store (APP)]
在 Android 與 iOS 專案中導入原生影音播放器庫 (如 ExoPlayer / AVPlayer) 以解析並播放 Mux 傳來的 HLS 流。
驗收標準 (Acceptance Criteria):
  • 支援全螢幕切換與子母畫面 (PiP) 模式，讓使用者可以邊看直播邊滑動商品。
[FE-402] 網頁端與行動端之即時聊天室與購物 UI
[前端專案: Potato-Store (Web & APP)]
開發覆蓋於播放器上的 Socket.io 聊天彈幕介面，並實作底部「熱賣商品」抽屜選單 (Bottom Sheet)。
驗收標準 (Acceptance Criteria):
  • 接收到「推廣事件」時，畫面上方需彈出商品卡片，點擊可直接加入購物車。
[FE-403] 網站載入效能最佳化 (Core Web Vitals)
[前端專案: 所有 Web 專案]
利用 Vite 配置 Code Splitting，壓縮圖片資產 (轉換為 WebP)，減少首屏 Bundle Size，提升 LCP 與 CLS 分數。
驗收標準 (Acceptance Criteria):
  • Google Lighthouse 首頁效能評分必須達到 85 分以上。
[FE-404] 前端錯誤收集與監控 (Sentry/Application Insights) 整合
[前端專案: 所有 Web & APP]
導入錯誤追蹤 SDK，捕獲前端 JavaScript 崩潰與 API 非預期例外錯誤，統一發送至儀表板。
驗收標準 (Acceptance Criteria):
  • 發生 Error Boundary 捕獲的崩潰時，能附帶使用者的 Session 與 Browser 資訊回傳至後台。
