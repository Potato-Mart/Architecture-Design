第八梯次：VR 版商場體驗

目標期限：2027年下半年 | 重點：WebXR 網頁先鋒版、3D 建模與互動體驗

階段任務 核心目標與範圍

導入全新的元宇宙/虛擬實境概念，提升消費者互動體驗與品牌科技感。

  • 基於瀏覽器的 VR：採用 WebXR 與 Three.js 實作網頁版虛擬展間。
  • 行動端擴增：視網頁版進度，逐步將 3D 模型檢視能力 (AR/VR) 整合入原生 APP。
  • 沈浸式購物：將原有的 Commerce API 橋接至 3D 介面中，實現點擊虛擬商品直接加入購物車。

潛在風險

  • 3D 模型 (GLTF/OBJ) 檔案過大，導致網頁載入時間過長，流失使用者。
  • 手機端 WebGL 效能不足導致嚴重的卡頓或設備發燙。
  • VR 購物動線設計不良，反而降低結帳轉換率。

修復與改進

  • 模型資產進行抽殼與多邊形減面 (Decimation) 處理，並嚴格利用 Azure Blob Storage 的 CDN 邊緣節點進行快取。
  • 設計 2D/3D 切換按鈕，若偵測到設備效能不足，自動降級 (Graceful Degradation) 回 2D 介面。

後端工程 Backend Backlog Tickets

[BE-801] 3D 模型與素材管理 API
[微服務: Ops API (Go)]
擴充產品型錄功能，支援上傳 `.gltf` 與 `.usdz` 檔案至 Azure Blob Storage，並於 MongoDB Atlas 關聯。
驗收標準 (Acceptance Criteria):
  • 上傳 API 需驗證檔案副檔名與大小 (不得超過 20MB)。
  • 回傳的 API Response 中需包含 `model_3d_url` 欄位供前端取用。
[BE-802] 擴充 BFF 層以支援場景初始化資料 (Scene Data)
[微服務: Gateway BFF (Node.js)]
為 WebXR 網頁提供專屬的場景初始化 API，一次性回傳虛擬展間座標矩陣 (Transform Matrix) 與關聯商品。
驗收標準 (Acceptance Criteria):
  • 資料需經過高度壓縮與 Redis 快取，回應時間需極低。
[BE-803] 實作 VR 空間中的多人互動/看見彼此 (PoC)
[微服務: Live Stream API (Node.js)]
(可選) 擴展原有的 WebSocket 服務，廣播同一展間內其他消費者的簡易座標 (Avatar Position)，打造元宇宙的共鳴感。
驗收標準 (Acceptance Criteria):
  • 位置廣播必須具備節流 (Throttling) 機制，例如每秒只廣播 5 次座標，避免壓垮伺服器。
[BE-804] 3D 模型轉檔排程 Worker
[微服務: Ops Worker (Go)]
撰寫背景工作腳本，當使用者上傳高精度的原始 3D 模型時，在背景自動呼叫第三方 API 或命令列工具進行壓縮減面。
驗收標準 (Acceptance Criteria):
  • 自動產生一個小於 5MB 的 Web-optimized 版本供網頁端載入。

前端工程 Frontend Backlog Tickets

[FE-801] WebXR 虛擬展間搭建 (Three.js / React Three Fiber)
[前端專案: Potato-Store (Web)]
導入 `@react-three/fiber`，在網頁上渲染出 3D 商場場景，實作第一人稱視角移動邏輯。
驗收標準 (Acceptance Criteria):
  • 在配備中高階顯卡的電腦上，需維持穩定的 60 FPS 幀率。
[FE-802] 實作 3D 物件點擊與「加入購物車」互動
[前端專案: Potato-Store (Web)]
利用 Raycaster 偵測使用者的滑鼠點擊/螢幕觸控，當擊中 3D 產品時跳出該商品的詳細規格與價格。
驗收標準 (Acceptance Criteria):
  • 點擊 3D 空間中的「購買」按鈕後，需能正確更新外部 React 元件的 Zustand 購物車狀態。
[FE-803] 原生 APP 整合 ARKit / ARCore (AR 檢視)
[行動端: Potato-Store (APP)]
在 Android 與 iOS 平台中，開發「在您的空間預覽」功能，呼叫原生的 AR 套件檢視實體大小。
驗收標準 (Acceptance Criteria):
  • iOS 成功載入 `.usdz` 檔案並喚起 Quick Look。Android 成功載入 `.gltf` 檔案。
[FE-804] VR 模式效能降級與防呆處理
[前端專案: Potato-Store (Web)]
實作進場前的硬體偵測，以及加載 3D 資源時的百分比進度條 (Loading Screen)。
驗收標準 (Acceptance Criteria):
  • 若瀏覽器不支援 WebGL 或效能過低，必須平滑引導使用者回到標準 2D 商品列表頁面。
