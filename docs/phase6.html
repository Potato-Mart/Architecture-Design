第六梯次：對外開放系統 (Open API)

目標期限：2027年5月底前 | 重點：升級 Azure API Management、外部開發者入口

階段任務 核心目標與範圍

開放系統核心能力，讓外部夥伴、物流供應商或其他企業能以程式化方式對接我們的電商與庫存資料。

  • 升級基礎設施：正式導入企業級的 Azure API Management (APIM) 作為對外防護閘道。
  • 實作 OAuth 2.0 (Client Credentials) 以核發存取金鑰 (API Key)。
  • 建立 Developer Portal (開發者入口網站)，提供詳細的 API 文件 (Swagger/OpenAPI)。

潛在風險

  • 開放對外 API 若缺乏完善的限流與配額管理，可能遭受 DDoS 攻擊。
  • 外部開發者呼叫 API 造成資料毀損或越權存取他人資料。
  • APIM 服務訂閱與計費模型若設計不當，可能導致營收損失。

修復與改進

  • 透過 Azure API Management 實施嚴格的 Subscription Key 與速率限制 (Rate Limiting)。
  • 提供隔離的 Sandbox (測試環境) 供合作夥伴開發測試。
  • 在 APIM 層實作 Request/Response 驗證，阻擋不符合 OpenAPI 規範的 payload。

後端工程 Backend Backlog Tickets

[BE-601] 正式導入 Azure API Management 基礎架構
[基礎設施: Azure APIM]
部署 Azure API Management，設定將外部 API 路由轉發至內部的 Azure Container Apps 微服務叢集。
驗收標準 (Acceptance Criteria):
  • 所有來自外部的直接存取內部微服務請求必須被擋下，只允許經由 APIM 的流量。
[BE-602] 實作 APIM 限流政策 (Policies) 與金鑰管理
[基礎設施: Azure APIM]
撰寫 APIM Policies XML，針對不同等級的訂閱帳號設定不同的 API 呼叫配額 (Quota) 與速率限制 (Rate limit)。
驗收標準 (Acceptance Criteria):
  • 免費測試帳號每分鐘限制 60 次呼叫，超過回傳 `429 Too Many Requests`。
[BE-603] OpenAPI (Swagger) 文件標準化與自動生成
[微服務: 全體 Go 微服務]
在 Go 程式碼中補齊 Swag 註解，由 CI/CD 流程自動產生 OpenAPI Specification JSON 並同步更新至 APIM 的 Developer Portal。
驗收標準 (Acceptance Criteria):
  • 每個開放的 Endpoint 必須附帶 Request Example 與 Response Schema 說明。
[BE-604] 實作外部 Webhook 訂閱與推送機制
[微服務: Ops API (Go) / Event Hubs]
允許外部夥伴註冊他們的 URL Endpoint，當「訂單狀態更新」或「庫存變動」時，系統主動打 Webhook 通知對方。
驗收標準 (Acceptance Criteria):
  • 發送 Webhook 時需附帶 HMAC 簽章讓夥伴驗證來源。
  • 若對方伺服器無回應，需具備 Exponential Backoff 重試機制。

前端工程 Frontend Backlog Tickets

[FE-601] 開發者入口網站 (Developer Portal) 佈景客製化
[前端專案: Azure APIM Portal]
利用 Azure APIM 內建的開發者入口網站，套用土豆商城的品牌視覺 (CSS) 與排版，提供無縫的體驗。
驗收標準 (Acceptance Criteria):
  • 入口網站需具備自適應網頁設計 (RWD)，且色彩對比度符合無障礙規範 (WCAG)。
[FE-602] 後台新增「開發者金鑰管理」與「配額檢視」介面
[前端專案: Potato-Control (Web)]
在內部營運後台中，新增管理頁面以審核外部廠商的 API 申請，並視覺化顯示他們的 API 呼叫用量 (Chart.js)。
驗收標準 (Acceptance Criteria):
  • 管理員可以一鍵撤銷 (Revoke) 異常暴增流量的金鑰。
[FE-603] 實作互動式 API 測試終端 (API Console)
[前端專案: Developer Portal]
在開發者入口整合 Swagger UI 或類似套件，讓開發者能在網頁上直接填入參數發送測試請求。
驗收標準 (Acceptance Criteria):
  • 測試請求需預設帶入使用者的測試金鑰 (Sandbox Key)，不可誤觸正式環境。
[FE-604] 外部 Webhook 註冊與日誌檢視 UI
[前端專案: Potato-Partner (Web)]
在 B2B 的企業後台中，新增區塊讓外部廠商自行輸入 Webhook 接收網址，並顯示最近 100 筆推送紀錄與成功/失敗狀態。
驗收標準 (Acceptance Criteria):
  • 需具備重新發送 (Retry) 單筆 Webhook 事件的按鈕。
