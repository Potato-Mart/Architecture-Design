第五梯次：可規模化複製系統 (SaaS 化)

目標期限：2027年2月底前 | 重點：多租戶架構 (Multi-tenancy)、新廠區與門市快速展店

階段任務 核心目標與範圍

重構現有資料庫與應用層邏輯，將平台轉型為類 SaaS 模式，以支援未來業務的快速擴張與展店需求。

  • 系統升級為「可規模化複製 (Scalable & Replicable)」，能快速部署給新的廠區與實體門市使用。
  • 提供多租戶 (Multi-tenant) 管理，支援獨立的庫存、員工與帳務結算。
  • 利用 MongoDB Atlas 分區能力，建立「總部 (Global) 與 門市 (Local)」的分級設定中心。

潛在風險

  • 租戶資料隔離不夠嚴謹，可能導致 A 門市看到 B 門市的訂單或營收 (Cross-tenant Data Leak)。
  • 全域庫存 (總倉) 與區域庫存 (門市) 的調撥邏輯複雜，容易出現庫存對不齊的 Bug。

修復與改進

  • 在業務邏輯層強制注入 `tenant_id` 的嚴格過濾，確保 MongoDB Atlas 租戶資料絕對隔離。
  • 建立階層式權限架構與全域/區域參數的覆寫 (Override) 機制。
  • 所有內部 API 路由加入 `/{tenant_id}/` 作為基礎前綴，並由 Gateway/Ingress 阻擋非法越權。

後端工程 Backend Backlog Tickets

[BE-501] 資料庫 Multi-tenant 綱要重構與遷移
[基礎設施: MongoDB Atlas / Go]
更新所有 Collections (Orders, Inventories, Staff) 加入強制的 `tenant_id` 欄位。撰寫 Migration 腳本將現有資料歸屬至預設的第一間門市 (Tenant 1)。
驗收標準 (Acceptance Criteria):
  • 所有資料表讀寫操作，若未傳入 Tenant ID，必須噴出 Panic 或拒絕執行。
[BE-502] 實作全域與區域 (Global vs Local) 庫存調撥邏輯
[微服務: Ops API (Go)]
開發跨租戶的庫存轉移 (Inventory Transfer) API，支援總倉調配庫存至各分店，並記錄詳細的調撥歷程。
驗收標準 (Acceptance Criteria):
  • 調撥過程需使用 MongoDB Transaction，確保出庫與入庫的數字一致且原子性。
[BE-503] 建立自動化新租戶開通腳本 (Tenant Onboarding)
[基礎設施: DevOps / Go]
開發一套自動化流程，當新門市或新廠區成立時，一鍵生成預設角色、設定檔、並開通獨立的資源與報告。
驗收標準 (Acceptance Criteria):
  • 開通新廠區的時間需小於 30 秒，且新管理員能立即登入並擁有該廠區最高權限。
[BE-504] API 路由與 JWT 驗證層級改造
[微服務: Auth API (Go)]
更新 JWT 的 Payload，使其包含該員工所屬的授權廠區陣列 `allowed_tenants: [1, 2]`。更新 Middleware 進行比對。
驗收標準 (Acceptance Criteria):
  • 若員工嘗試透過 API 存取未授權廠區的資料，必須嚴格阻擋並回傳 403 Forbidden。

前端工程 Frontend Backlog Tickets

[FE-501] 總部與門市上下文切換 (Context Switcher) UI
[前端專案: Potato-Control (Web & APP)]
在後台導覽列實作一個下拉式選單，允許具備多廠區權限的區主管/總經理，快速切換當前的操作租戶 (Tenant Context)。
驗收標準 (Acceptance Criteria):
  • 切換租戶後，全局狀態 (Zustand/Context) 更新，並自動清除舊租戶的 React Query 快取，觸發畫面重新渲染。
[FE-502] 多租戶資料整合數據看板 (Multi-tenant Dashboard)
[前端專案: Potato-Control (Web)]
開發總經理專用的全域儀表板，整合各廠區的營收、庫存健康度與熱銷商品排行，使用 Recharts 或 Chart.js 進行視覺化。
驗收標準 (Acceptance Criteria):
  • 支援圖表鑽取 (Drill-down)，點擊特定門市柱狀圖可跳轉至該門市的詳細數據頁面。
[FE-503] POS APP 綁定廠區與裝置註冊流程
[行動端: Potato-Control (APP)]
在新廠區啟用新平板時，實作初次開機的「綁定廠區/裝置註冊」精靈 (Wizard)，掃描 QR Code 即可自動帶入對應的 Tenant ID 與設定檔。
驗收標準 (Acceptance Criteria):
  • 綁定成功後需將配置鎖死在設備中，一般員工無法任意解除綁定或切換。
[FE-504] B2C 與 B2B 客戶端支援多門市自取選擇
[前端專案: Store & Partner (Web/APP)]
修改購物結帳流程，若使用者選擇「實體店自取」，能列出附近有庫存的廠區/門市，並顯示該店的營業時間。
驗收標準 (Acceptance Criteria):
  • 若該門市特定商品缺貨，該門市選項須標示為反灰 (Disabled)，並提示「庫存不足」。
