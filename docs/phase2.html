第二梯次：替換現有系統與 POS 串接

目標期限：7月底前 | 重點：商業結帳、Adyen 金流與舊系統退役

階段任務 核心目標與範圍

完全淘汰現有的 Pisell 系統，並整合全新的線下實體店面結帳體驗與 Azure 佇列。

  • 完整商業模組：購物車、結帳流程、訂單狀態機。
  • 實體門市整合：開發原生平板 APP (Potato-Control) 導入新 POS 機。
  • 金流處理：完成 Adyen POS 支付金流串接。
  • 異步處理：引入 Azure Event Hubs 作為 Webhook 與結帳重試佇列。

潛在風險

  • 實體門市網路不穩導致 POS 機與 Adyen 刷卡機斷線。
  • Adyen API 回應延遲、Webhook 漏發，導致訂單狀態卡在「付款中」。
  • Pisell 舊有資料格式不相容導致寫入 MongoDB Atlas 失敗。

修復與改進

  • 在 Potato-Control APP 中實作離線模式 (Offline-first)，網路恢復後自動同步。
  • 後端導入 Azure Service Bus 或 Event Hubs 佇列，確保交易事件百分百送達。
  • 撰寫專門的 ETL (Extract, Transform, Load) 腳本處理 Pisell 資料清理。

後端工程 Backend Backlog Tickets

[BE-201] 開發 Commerce 領域 API (購物車與結帳)
[微服務: Commerce API (Go)]
實作建立訂單、更新購物車、計算運費與稅金之 API，並處理併發下單的問題。
驗收標準 (Acceptance Criteria):
  • API 能正確計算商品小計、稅金與折扣。
  • 使用 MongoDB 的分散式 Transaction，確保扣庫存與建立訂單必須同時成功或失敗。
[BE-202] 整合 Adyen POS 金流與 Webhook 佇列
[微服務: Commerce API (Go) / Azure Service Bus]
實作安全的 Webhook 端點以接收 Adyen 非同步付款結果，並將 Payload 直接放入 Azure Service Bus Queue。
驗收標準 (Acceptance Criteria):
  • 能成功驗證來自 Adyen 的 HMAC 簽章，防止偽造請求。
  • Webhook 請求需在 2 秒內回覆 200 OK，後續交由 Worker 從 Service Bus 取出慢慢處理，處理失敗需轉移至 Dead-Letter Queue。
[BE-203] 實作 POS 機離線資料同步端點 (Bulk Sync API)
[微服務: Commerce API (Go)]
提供批次上傳 (Bulk upload) API，讓實體店 POS 機重新連網時能一次性上傳所有未結訂單。
驗收標準 (Acceptance Criteria):
  • API 必須支援接收 JSON Array，並以 MongoDB `insertMany` 或 `bulkWrite` 的方式寫入，避免佔用連線池。
[BE-204] Pisell 資料轉移 ETL 腳本開發
[工具程式: Data-Migrator (Python / Go)]
撰寫腳本將舊系統的 SQL/CSV 資料匯出，清理並轉換成符合新 MongoDB Atlas 綱要的格式後寫入。
驗收標準 (Acceptance Criteria):
  • 會員的密碼雜湊能夠無縫轉移，確保舊會員不需強制重設密碼。
  • 歷史訂單資料成功匯入，並且在前後端系統均可正確渲染。

前端工程 Frontend Backlog Tickets

[FE-201] 搭建 potato-store 顧客前台網站
[前端專案: Potato-Store (Web)]
使用 Vite + React + TypeScript 框架，實作首頁、商品列表與購物車的 State Management (Zustand)。
驗收標準 (Acceptance Criteria):
  • 購物車資料需同步儲存於 localStorage，重新整理頁面後不流失。
[FE-202] 實作結帳表單與 Adyen Drop-in UI
[前端專案: Potato-Store (Web)]
開發包含填寫收件地址、選擇運送方式的結帳流程，並嵌入 Adyen Web Drop-in 元件供線上刷卡。
驗收標準 (Acceptance Criteria):
  • 使用者成功刷卡後，需正確跳轉至「付款成功」頁面並清除購物車。
  • 若信用卡被拒絕，需顯示對應的友善錯誤提示。
[FE-203] 搭建 potato-control POS 機 APP (Expo/React Native)
[行動端: Potato-Control (APP)]
使用 Expo + React Native + NativeWind 建立平板專用 POS 介面。配置 Expo Router 管理路由。
驗收標準 (Acceptance Criteria):
  • APP 需針對 iPad/Android 平板等大螢幕設備進行橫向 (Landscape) UI 最佳化。
[FE-204] 實作 POS 機 Offline-first 結帳邏輯
[行動端: Potato-Control (APP)]
在 POS APP 中，使用 AsyncStorage 暫存斷線時的訂單，並實作背景同步 (Background Sync)。
驗收標準 (Acceptance Criteria):
  • 拔除平板網路後，店員仍能點選商品並建立本機訂單。
  • 網路恢復後，系統會自動在背景呼叫 [BE-203] 進行批次同步，並提供成功 Toast 提示。
